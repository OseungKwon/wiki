## 정의
로그인 시 클라이언트가 비밀번호(및 아이디)를 서버로 보내기 전 암호화하는 방식. HTTPS 위에 얹는 응용 레벨 암호화로, 대칭키(AES)와 비대칭키(RSA) 두 갈래가 있다. 핵심 차이는 **"복호화 키를 클라이언트가 갖느냐"** 이며, 이것이 보안 강도와 규제 대응을 가른다.

## 상세 설명

### 대칭 vs 비대칭 — 근본 차이
- **대칭키(Symmetric)**: 암호화·복호화에 **같은 키**를 쓴다. 빠르고 가볍지만, 양측이 같은 비밀(키)을 공유해야 한다. → "키를 어떻게 안전하게 공유하느냐(키 분배 문제)"가 본질적 약점.
- **비대칭키(Asymmetric)**: 수학적으로 짝지어진 **공개키/개인키** 쌍을 쓴다. 공개키로 암호화하면 개인키로만 복호화된다. 키 분배 문제가 없다(공개키는 공개해도 됨). 대신 느리고 처리 가능한 데이터 크기가 작다.

| 구분 | AES (대칭) | RSA (비대칭) |
|------|-----------|-------------|
| 키 | 양측이 같은 키 공유 | 공개키(암호화) / 개인키(복호화) 분리 |
| 클라이언트 보유 키 | 복호화 가능한 키 | **공개키만** (복호화 불가) |
| 속도 | 빠름 (대량 데이터에 적합) | 느림 (작은 데이터에만) |
| 핸드셰이크 | 불필요 | 공개키 전달 필요(임베드 or 발급) |
| 비밀번호 노출 위험 | 번들에서 키 추출 시 복호화 가능 | 개인키 없으면 복호화 불가 |

### AES (Advanced Encryption Standard) — 대칭키 상세
- **블록 암호**: 평문을 16바이트(128비트) 블록 단위로 암호화한다. 키 길이에 따라 AES-128/192/256으로 나뉜다(숫자는 키 비트 수, 블록 크기는 항상 128비트).
- **운영 모드(Mode)**: 블록들을 어떻게 연결하느냐의 규칙.
  - **CBC (Cipher Block Chaining)**: 각 블록을 직전 암호문과 XOR한 뒤 암호화. 첫 블록엔 **IV(초기화 벡터)** 가 필요. 동일 평문이라도 IV가 다르면 다른 암호문이 나온다. 단 **무결성(변조 탐지) 보장 없음**.
  - **GCM (Galois/Counter Mode)**: 기밀성 + **무결성/인증(AEAD)** 을 함께 제공. 변조 시 복호화가 실패한다. 현대 권장 모드. (`crypto-js`는 GCM 지원이 약해 Web Crypto API 사용 권장)
- **IV(초기화 벡터)**: 같은 키로 같은 평문을 암호화해도 매번 다른 암호문을 만들기 위한 난수 블록(16바이트). **요청마다 랜덤**이어야 하며 암호문과 함께 전송한다(비밀이 아님). 고정 IV는 "같은 평문 → 같은 암호문" 패턴을 노출시켜 약점이 된다.
- **패딩(Padding)**: 평문 길이가 블록 크기의 배수가 아닐 때 채우는 규칙. 표준은 **PKCS7**. 비표준 커스텀 패딩은 상호운용·유지보수 리스크만 늘리고 보안 이득이 없다.
- **약점의 본질**: AES 자체는 안전하지만, **키를 안전하게 보관·공유하지 못하면 무의미**하다. 클라이언트 번들에 키를 박으면 누구나 추출 가능 → 사실상 암호화 효과 0.

### RSA — 비대칭키 상세
- **수학적 기반**: 큰 두 소수의 곱(modulus `n`)에 대한 소인수분해의 어려움에 기반. 공개키 = `(n, e)`, 개인키 = `(n, d)`. 공개키로 암호화(`c = m^e mod n`)하면 개인키로만 복호화(`m = c^d mod n`)된다.
  - `e`(공개 지수)는 보통 **65537(0x10001)** 을 쓴다. 배민 응답의 `value:"10001"`이 바로 이 값.
  - `n`(modulus)의 비트 수가 곧 RSA 키 길이. **RSA-2048 이상** 권장(RSA-1024는 현재 기준 미흡).
- **패딩(Padding)**: RSA도 패딩이 필수다. 패딩 없는 raw RSA는 안전하지 않다.
  - **PKCS#1 v1.5**: 구식. 패딩 오라클 공격 취약성 이력이 있어 신규 설계엔 비권장.
  - **OAEP**: 현대 권장 패딩. 신규 구현은 **RSA-2048 + OAEP** 조합을 쓴다.
- **크기 제약**: 한 번에 암호화 가능한 데이터는 키 크기보다 작다(RSA-2048 + OAEP ≈ 최대 190바이트). 비밀번호 같은 짧은 데이터엔 충분하지만, 큰 데이터는 못 다룬다 → 그래서 큰 데이터엔 하이브리드(아래) 방식을 쓴다.
- **강점의 본질**: 클라이언트는 **공개키만** 가지므로 번들이 털려도 복호화는 불가능하다. "클라이언트에 복호화 키가 없다"는 점이 규제 대응의 핵심.

### 하이브리드 (RSA + AES)
TLS가 쓰는 방식. 데이터는 빠른 **AES**로 암호화하고, 그 **AES 키만 RSA로 감싸서(암호화해서)** 전달한다. "대칭의 속도 + 비대칭의 안전한 키 분배"를 모두 취한다. 다만 비밀번호처럼 작은 데이터엔 RSA 단독으로 충분하므로 하이브리드는 과하다.

### 흔한 안티패턴 — 하드코딩 고정키 AES
클라이언트 번들에 AES 키/IV를 하드코딩하는 방식. 번들만 디컴파일하면 키를 추출해 누구나 복호화/재암호화가 가능하므로 **기밀성 기여도는 사실상 0**. 실제 보호는 HTTPS가 담당하고, 이 암호화는 "네트워크 탭/로그에 평문이 안 찍히는" obfuscation 수준에 그친다.

```ts
// 안티패턴: 키와 IV가 번들에 노출됨
const key = 'VKDOWHGMDK...';   // 하드코딩
const iv  = 'ZZXOCMDK...';     // 하드코딩 (고정 IV → 같은 평문은 항상 같은 암호문)
CryptoJS.AES.encrypt(text, SHA256(key), { iv: SHA256(iv), mode: CBC });
```

### 랜덤 IV는 해결책이 아니다
고정 IV → "같은 비밀번호는 항상 같은 암호문" 패턴 노출이라는 **하위 문제**만 랜덤 IV로 해결된다. 키가 번들에 박혀 있는 **근본 문제**는 그대로이므로, 감사 지적("하드코딩 키")은 닫히지 않는다. 랜덤 IV는 "해야 할 위생 조치"지 "이슈 해결책"이 아니다.

### "AES + 서버 발급 키"의 함정
서버가 AES 키를 발급해도, 그 키를 **평문으로 응답에 담으면** 키가 암호문과 같은 채널(HTTPS)을 타므로 보안 이득이 없다(지적이 "응답에 키 노출"로 이동할 뿐). 제대로 하려면 키를 RSA로 감싸 전달해야 하는데, 이는 곧 하이브리드로 **RSA 단독보다 복잡**해진다. 비밀번호처럼 짧은 데이터엔 하이브리드의 이점이 없다.

## 실무 적용

### 사례 비교
- **배민 비즈회원 로그인**: `/v1/login/init`에서 RSA 공개키(modulus `tag`, exponent `value=10001`) 수신 → 아이디/비밀번호를 RSA 암호화해 `value1`/`value2`로 전송. (단 RSA-1024 + PKCS#1 v1.5라 키 길이/패딩은 요즘 기준 미흡)
- **passorderboss-web (현재)**: `crypto-js`로 AES-256-CBC, 키/IV 하드코딩 + 고정 IV + 커스텀 패딩. `POST /auth/authentications/login`에 암호화된 password 전송.

### 전자금융업 대응 결론
전자금융감독규정·금융보안원 가이드는 인증정보의 전송구간 암호화 + 안전한 키관리를 요구하며, 감사 지적의 본질은 대개 "키 노출"이다.

| 옵션 | 지적 해결 | 서버 변경량 | 비밀번호용 적합성 |
|------|:---:|:---:|:---:|
| 랜덤 IV / AES 유지 | ❌ | 소 | 위생만 |
| AES + 서버 발급 키(평문) | ❌ | 중 | 무의미 |
| AES + 서버 발급 키(RSA 래핑) | ✅ | 대 | 과함 |
| **RSA 단독 (2048+OAEP)** | ✅ | 중 | **최적** |

→ **RSA 단독이 정답.** 서버 변경은 "개인키 안전 보관(KMS/HSM 권장) + decrypt 함수 교체"로 재작성이 아님. 복호화 이후 흐름(평문 비번 검증)은 동일.

### 마이그레이션(혼재) 처리
신·구 클라이언트가 섞여도 서버가 포맷을 분기하면 안전하다. Base64 알파벳에 없는 구분자/버전 프리픽스(`v2:base64(iv):base64(ct)`)로 신·구 포맷을 100% 구분 가능. 단 IV뿐 아니라 **패딩 방식도 포맷별로 함께 분기**해야 한다.
